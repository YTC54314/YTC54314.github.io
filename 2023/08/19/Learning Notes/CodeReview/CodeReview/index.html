Code Reviewer 的心法與職權邊界

2023-08-19
2023-10-09
code review

目的

  1. 符合團隊規範：架構、風格
  2. 程式品質
  3. 測不出來的東西，例如：維護性、可讀性
  4. 邏輯漏洞（非功能正確性）、安全漏洞
  5. 知識分享交流（自己可以學習）

心法、準則

  1. Done is better than perfect (scrum 精神)。
  2. 功能有達到就60分了，我們目標是80分，不要求100分。
  3. 大架構到程式碼
    1. 掃過一眼程式的檔案結構，是否有不該上傳或缺漏的檔案
    2. 命名有沒有意義/不一致
    3. 重複的程式碼有沒有定義成 function
    4. 冗長的程式碼能不能拆分 function
    5. 命名有沒有意義/不一致
  4. 越底層、之後修改成本高的優先
    1. 資料庫設計
    2. 架構規劃
  5. 迭代性review，建議著重在 3 – 5 個優化地方，比較容易聚焦在優化的品質。不然每次都看這麼多問題兩邊都很累，每次都進步一點大於一堆問題沒有結論。根據時程的壓力，決定 Code Review 迭代的次數。
  6. 不要惡化原有系統就好，每修正到一個項目對系統來說是bonus。
  7. 功能性錯誤或粗心錯誤是最不希望發生的。

練習

  1. 聰明的 code review 也是門技術，希望不要過度吹毛求疵導致上版延遲，更不要讓review流於形式
  2. 迭代性review
  3. 不只抓錯，有學習到的地方也可以給予稱讚
  4. [blocker]、[optional]、[question]、［good］
    1. [blocker] 必須要調整，須優先處理或討論
    2. [optional] 有時間可以調整
    3. [question] 對於作法的疑問或好奇
    4. [good] 因為review而學習到的部分
  5. 正面回饋、有效溝通
  6. 當達成共識變得特別困難時，reviewer 和開發者可以進行面對面的會議，或者請小組長參與調停，而不僅僅是試著通過代碼審查評論來解決衝突。不要因為不能達成共識，就讓 PR 卡在那裡。
